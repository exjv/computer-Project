# 第1步：项目扫描与改造清单

## 1. 项目目录结构

### 后端（Spring Boot）
- 入口：`backend/src/main/java/com/jou/networkrepair/NetworkRepairApplication.java`
- 公共层：`common`（统一响应、异常、JWT、安全配置、日志切面）
- 模块层：
  - `module/auth`：登录、验证码、第三方登录预留
  - `module/user`：用户管理
  - `module/device`：设备管理
  - `module/repair`：报修工单、维修记录、流转、自动派单算法
  - `module/notice`：门户/公告
  - `module/log`：登录日志、操作日志

### 前端（Vue3 + Element Plus + Pinia + Vue Router）
- 核心：`frontend/src/main.js`
- 路由：`frontend/src/router/index.js`
- 状态管理：`frontend/src/stores/user.js`
- API：`frontend/src/api/index.js`
- 页面：
  - 认证：`views/auth`（门户、登录）
  - 首页：`views/dashboard/HomeView.vue`
  - 业务：`views/device`、`views/repair`、`views/record`
  - 管理：`views/user`、`views/log`、`views/notice`
  - 个人中心：`views/profile`

## 2. 当前已有模块能力评估

### 登录与认证
- 已有验证码登录、JWT 鉴权、角色选择登录入口。
- 已有门户页角色入口。
- 第三方 OAuth 仅为预留接口，未真实接入。

### 用户与权限
- 用户管理支持分页、增删改、启停、重置密码。
- 角色字段存在，但 RBAC 以“角色字符串 + 注解”实现，缺少资源级权限模型。

### 设备管理
- 设备支持基础 CRUD。
- 已有设备档案页接口 `/{id}/profile`，可查看关联工单/维修记录统计。
- 缺少设备生命周期（入库、启用、维护、停用、报废）状态机与约束。

### 报修工单
- 已有创建、审核、分配、接单、维修、验收、完成/取消等状态与流程记录。
- 已有自动派单算法（紧急度 + 负载）。
- 缺少 SLA / 预计修复时间自动计算与超时预警闭环。
- 缺少附件上传与图片留痕。

### 维修记录
- 已有维修记录 CRUD，基本与工单关联。
- 缺少标准化结案模板与知识库沉淀。

### 日志审计
- 已有登录日志、操作日志。
- 缺少审计检索条件、导出、异常行为分析与不可抵赖设计。

### 统计分析
- 首页已有概览卡片和图表（工单统计、设备状态、完成率）。
- 缺少多维统计（时间、部门、设备类型、维修人绩效）与报表导出。

## 3. 当前项目问题清单（针对目标系统）

1. **仍偏“增强型 CRUD”**：虽已有流程接口，但前后端仍存在大量通用增删改查交互，业务规则未完全收敛到领域服务。
2. **RBAC 不完整**：缺少菜单/按钮/接口级权限点、数据域隔离策略，当前多为角色硬编码。
3. **验证码登录已具备（✅）**，但缺少失败次数限制、账号锁定、风控策略。
4. **角色入口页已具备（✅）**，但缺少角色门户（分角色待办、指标、快捷操作）差异化首页。
5. **工单生命周期基本具备（✅）**，但缺少 SLA、超时、催单、升级、关闭原因标准化。
6. **设备详情页已具备（✅）**，但缺少生命周期履历（变更、巡检、保修到期、报废审批）。
7. **日志审计基础具备（✅）**，但检索/追溯能力弱，且操作日志用户名取值依赖 principal，准确性需加强。
8. **统计联动较弱（⚠️）**：仅首页聚合，缺少专门统计中心与筛选联动。
9. **文件上传与图片留痕缺失（❌）**：报修/维修过程无法上传现场图片、日志附件。
10. **报表导出缺失（❌）**：无 Excel/PDF 导出接口与页面按钮。
11. **维修推荐分配已有基础（✅）**，但未引入技能标签、区域班次、历史成功率等更真实约束。
12. **预计修复时间字段存在但流程使用不足（⚠️）**：`expected_finish_time` 未形成自动计算、展示和预警闭环。

## 4. 需要新增/修改的数据表清单

### 新增表
1. `sys_role`、`sys_permission`、`sys_role_permission`、`sys_user_role`（从角色字符串升级到可配置 RBAC）
2. `repair_order_attachment`（工单附件：报修图片/日志/验收图片）
3. `repair_record_attachment`（维修过程附件）
4. `device_lifecycle_log`（设备生命周期事件：入库、启用、维修、停用、报废）
5. `maintainer_skill`（维修人员技能标签）
6. `maintainer_schedule`（排班/值班信息）
7. `report_export_task`（异步导出任务）
8. `repair_sla_rule`（SLA 规则配置）
9. `repair_order_sla_log`（SLA 计算、超时、升级记录）

### 修改表
1. `repair_order`：补充 `campus_area`、`building`、`room`、`fault_images`（建议拆附件表）、`sla_deadline`、`dispatch_strategy`、`dispatch_reason`、`urgency_score`、`closed_by` 等。
2. `repair_record`：补充 `root_cause_category`、`solution_category`、`labor_hours`、`cost_amount`。
3. `network_device`：补充 `asset_status`、`warranty_expire_date`、`importance_level`、`maintain_vendor`。
4. `operation_log`：补充 `request_params`、`response_code`、`trace_id`。
5. `login_log`：补充 `user_agent`、`fail_reason`。

## 5. 需要新增/修改后端接口清单

### 设备生命周期
- 新增：设备生命周期事件接口、报废审批接口、设备健康度评分接口。

### 统计与报表
- 新增：统计中心接口（按时间、部门、设备类型、人员、校区维度）。

### 审计日志
- 修改：日志分页增加筛选条件（时间、模块、人员、状态）。

## 7. 分阶段实施计划

### 第一阶段：基础重构
- 目标：统一领域模型、状态机校验、异常码、日志追踪。
- 产出：流程状态机组件、统一权限拦截、DTO 校验补齐。

### 第二阶段：登录与门户
- 目标：完善认证风控与角色门户。
- 产出：验证码增强、账号锁定、分角色首页聚合接口与页面。

### 第三阶段：核心业务流程
- 目标：打通真实工单全流程。
- 产出：提交→审核→分配→接单→维修→验收→结案闭环；催办/升级/SLA。

### 第四阶段：设备与维修管理
- 目标：构建设备全生命周期与维修知识沉淀。
- 产出：设备履历、维修留痕、附件体系、维修结案模板。

### 第五阶段：统计分析与报表
- 目标：形成可管理决策的数据能力。
- 产出：多维统计中心、图表联动、报表导出（异步任务）。

### 第六阶段：用户与日志管理
- 目标：完善组织协作与审计追溯。
- 产出：可配置 RBAC、数据权限、审计检索与导出、操作追踪。
